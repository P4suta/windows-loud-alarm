# Build perf baseline — 2026-05-28 (after Phase 5)

Phase 5 (今 session) では計測 infrastructure (Phase 1–4 で完成) の上にビルド本体の最適化を載せた。
出発点と到達点を表で示し、 各 sub-phase の per-step 計測 md を `docs/perf/baseline-2026-05-28-phase5-*.md` に残してある。

|--------------------|---------------------:|-------------:|--------------:|------------------:|-----------------:|
| `cold-rebuild`     |               17.331 |       18.032 |        17.619 |        **16.954** | -0.38 (**-2.2 %**) |

Phase 5 で動いた主因は test 側 — xUnit v3 + Microsoft.Testing.Platform 直叩き化が
test-fast を 2.55 → 2.35 s に削った (-7.6 %)。 build 側 (`UseSharedCompilation`,

ノイズ範囲で、 計測精度の限界。 cold-rebuild の +1.7 % は per-run 揺れに収まる。

## What changed in Phase 5

### 5.0 — binlog 詳細解析 (`scripts/AnalyzeBinlog.cs`)
- `MSBuild.StructuredLogger@2.2.386` (NuGet) を file-based program に `#:package` で依存追加。
  Central Package Management 制約のため `Directory.Packages.props` に `PackageVersion` を登録、
  `#:package` は version 省略形に。
- `BinaryLog.ReadBuild()` で artifacts/perf/build.binlog を読み、 Target / Task / Project
  ごとの elapsed を集計、 Top 20 を markdown 出力。 `just analyze-binlog` でいつでも再実行。

### 5.1 — high-confidence wins
- `Directory.Build.props`: `<UseSharedCompilation>true</UseSharedCompilation>` を明示。
  (.NET 9+ default だが SDK regression 防護)
- `mise.toml`: `[env] MSBUILDUSESERVER = "1"` で persistent MSBuild server を有効化。
- `Directory.Build.props`: `<GenerateDocumentationFile Condition="'$(Configuration)' == 'Release'">true</GenerateDocumentationFile>`
  で Debug build から XML doc 生成を外す。

### 5.2 — binlog-driven targeted fixes (該当なし、skip)
binlog 解析で見えた warm-incremental の主犯は **MarkupCompilePass1/2 が warm でも 800 ms 走っている** こと。
WindowsAppSDK 提供の XAML compiler の incremental 設計上の挙動で、 production code を壊さずに
直す手段がない。 SDK アップデートを待つ。 SubsetFonts は incremental skip が効いていて
無罪 (target ranking に出ない)。

### 5.3 — dev loop proxy (subjective として保留)
`dotnet watch` の startup を厳密に Stopwatch 化する proxy が、 watch のログ出力タイミングに
依存して再現性に欠ける。 一旦 documentation のみで保留。

### 5.4 — xUnit v3 / Microsoft.Testing.Platform 移行
- `Directory.Packages.props`: `xunit` (2.9.3) → `xunit.v3` (3.2.2)、 `xunit.runner.visualstudio` (3.0.1 → 3.1.5)。
- 各テスト `.csproj`: `<OutputType>Exe</OutputType>` + `<UseMicrosoftTestingPlatformRunner>true</UseMicrosoftTestingPlatformRunner>` を追加。
- xUnit v3 の新 analyzer `xUnit1051` を満たすため、 `DispatchAsync(..., TestContext.Current.CancellationToken)` で
  各テスト dispatch に CancellationToken を渡す。
- `mise.toml`: `TESTINGPLATFORM_TELEMETRY_OPTOUT=1` で MTP テレメトリ無効化。
- **重要**: `dotnet test` (VSTest 互換 layer) 経由だと逆に **遅く** なる (~3.5 s)。
  `justfile` の `test`/`test-fast` を MTP exe **直叩き** に変更:
  ```
  artifacts/bin/Alarm.Domain.Tests/debug/Alarm.Domain.Tests.exe --no-progress
  artifacts/bin/Alarm.Application.Tests/debug/Alarm.Application.Tests.exe --no-progress
  ```
  これで wall-clock 2.35 s。
- 並列化を `dotnet run scripts/RunTests.cs` で挟む実験もしたが、 file-based program の起動
  オーバーヘッドが parallelism gain を打ち消したため revert。 sequential が安定して速い。

## Where the wall-clock budget now goes (cold-rebuild ~17.6 s)

binlog 解析 (`scripts/AnalyzeBinlog.cs` 出力) から:

| Category | ms (warm build sample) | Notes |
|---|---:|---|
| MSBuild task recursion (project ref traversal) | ~3,600 | 各 csproj が他の csproj に依存するたびに走る。 6 projects × 4-5 calls。 |
| ResolveProjectReferences | ~800 | 同上。 SDK 内部。 |
| MarkupCompilePass1/2 (XAML) | ~800 | WindowsAppSDK の XAML compiler が warm でも走る。 SDK 待ち。 |
| Exec (XamlCompiler.exe / makepri / fonttools) | ~750 | 外部プロセス起動コスト。 SubsetFonts は incremental で skip 済み。 |
| その他 (Resolve assemblies, package assets, framework refs) | ~400 | SDK 内部、通常はキャッシュで軽い。 |

主要 bottleneck はもう **MSBuild + WindowsAppSDK の SDK 内部処理** で、 user code 側からの
最適化余地はほぼ尽きた。 残るは SDK アップデートか、 アプリの WinUI3 → 別フレームワーク移行のような
architectural decision のみ。

## Caveats

- 3 iterations では ±5 % 程度のノイズ。 個別 sub-phase の Δ がノイズ範囲のものは「効果なし」とは断定できないが、
  「効果が誤差より小さい」とは言える。
- `cold-rebuild` の iteration 1 がしばしば +2 〜 +10 s の outlier を出す (Defender scan か NuGet warm-up)。
  中央値で読む前提。
- xUnit v3 移行で `Alarm.Domain.Tests.exe` と `Alarm.Application.Tests.exe` の直接実行を `just test` のデフォルトに
  したため、 `dotnet test` を意図的に使うレシピは無くなった。 IDE の Test Explorer (VSTest 経由) は
  `xunit.runner.visualstudio` 3.1.5 が引き続き対応するので問題なし。
